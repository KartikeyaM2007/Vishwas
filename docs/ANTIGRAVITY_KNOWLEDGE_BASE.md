# ANTIGRAVITY KNOWLEDGE BASE

## Project Info
* **Project Name:** Community Hero: AI-powered hyperlocal community issue reporting, validation, tracking, and resolution platform (formerly CityPulse)
* **Hackathon Problem Statement:** Community Hero - Hyperlocal Problem Solver
* **Evaluation Criteria Focus:** Problem solving and impact (20%), Agent depth (20%), Innovation (20%), Google tech usage (15%). Priority is on Gemini-powered civic intelligence, community validation, impact dashboards, map/location workflows, and clean deployability.

## Core Capabilities Needed
1. Identifying and reporting multiple hyperlocal issues (not just potholes).
2. AI-powered issue categorization and routing using Google AI Studio / Gemini.
3. Geo-location and map-based issue tracking.
4. Community verification through confirm/upvote/duplicate reporting.
5. Real-time or near-real-time issue status tracking.
6. Impact dashboards for admins and communities.
7. Predictive insights (issue hotspots, recurring categories, high-risk areas).
8. Gamification for citizen engagement (points, badges, contributor scores).

## Current Architecture
* **Backend:** FastAPI, functioning as the core API hub. Integrates with Gemini 2.5 Flash for intelligent natural language processing, categorization, and SQL translation (Upgraded due to 1.5 deprecation). Connected to Supabase for PostgreSQL and Cloudinary for images.
* **Admin Frontend:** React + Vite SPA. Displays complaints on a Leaflet map, an impact dashboard/filtration table, and an NL query interface for data analytics and predictive insights.
* **Flutter App:** Mobile citizen app. Captures images and GPS to post multiple types of complaints, and includes community validation features.
* **Database:** Supabase PostgreSQL (`complaints` table). Schema natively includes `pgvector` extension for semantic embeddings.

## Environment Variables
Expected variables:
* `SUPABASE_URL`, `SUPABASE_KEY`
* `GEMINI_API_KEY` (Must start with AIza or AQ. for newer API keys)
* `CLOUD_NAME`, `CLOUD_API_KEY`, `CLOUD_API_SECRET`
* `VITE_API_BASE_URL` (Frontend)

*Note on env loading:* The backend uses a cascading dotenv loader to check `backend/Team-Try/.env` -> `backend/.env` -> `.env`. All variations are strictly ignored in `.gitignore` to prevent secret leakage.
